# Repository Documentation Guide

This guide explains how to customize and populate the repository architecture documentation with actual data from your GitHub repositories.

## Overview

You now have a complete documentation system with:
- **10 repository pages** with templates for all 9 OSC repositories
- **Tabbed interface** showing Services, Controllers, API Endpoints, and Architecture
- **Centralized overview page** linking to all repositories
- **Reusable component** (`RepoArchitecture`) for consistent formatting

## How to Customize Each Repository Page

### Step 1: Identify Your Services

For each repository, list all services in the `services` array:

```typescript
services: [
  {
    name: 'ServiceName',
    description: 'What this service does'
  },
  // Add more services...
]
```

### Step 2: Map Controllers to Services

For each controller, specify which services it uses and what endpoints it exposes:

```typescript
controllers: [
  {
    name: 'ControllerName',
    services: ['ServiceName1', 'ServiceName2'],
    endpoints: [
      {
        method: 'GET|POST|PUT|DELETE|PATCH',
        path: '/api/path',
        description: 'What this endpoint does'
      },
      // Add more endpoints...
    ]
  },
  // Add more controllers...
]
```

### Step 3: Update Database Information

Set the database field (optional):

```typescript
database: 'MongoDB / PostgreSQL - Description'
```

## Example: Populating a Backend Service

Let's say you're populating `repo-osc-be1-admin-profile.mdx`:

```typescript
export const repoData = {
  name: 'OSC BE1 - Admin Profile (censof/osc-be1-admin-profile)',
  description: 'Backend service for managing user profiles...',
  database: 'PostgreSQL - Users & Profiles Tables',
  services: [
    {
      name: 'UserService',
      description: 'Handles user creation, authentication, and management'
    },
    {
      name: 'RoleService',
      description: 'Manages roles and permissions'
    },
    {
      name: 'ProfileService',
      description: 'Manages user profile data'
    }
  ],
  controllers: [
    {
      name: 'UserController',
      services: ['UserService'],
      endpoints: [
        {
          method: 'GET',
          path: '/api/v1/users',
          description: 'Retrieve all users'
        },
        {
          method: 'POST',
          path: '/api/v1/users',
          description: 'Create a new user'
        },
        // ... more endpoints
      ]
    }
    // ... more controllers
  ]
};
```

## Tips for Success

### For Backend Services

- Include all REST endpoints (GET, POST, PUT, DELETE, PATCH)
- Map each controller to its services
- Document database tables/collections
- Include authentication/authorization requirements

### For Frontend Applications

- Document page/screen components
- List API calls and their purposes
- Include state management services
- Document UI component hierarchy

### For Infrastructure

- Document configuration files
- Include namespace/environment details
- List deployments and resources
- Document network policies

## API Endpoint Documentation

Each endpoint should include:

| Field | Value |
|-------|-------|
| **method** | HTTP method (GET, POST, PUT, DELETE, PATCH) |
| **path** | Full API path (e.g., `/api/v1/users/:id`) |
| **description** | Brief description of what it does |

## Extending the Documentation

### Adding Query Parameters

Extend the endpoint object:

```typescript
endpoint: {
  method: 'GET',
  path: '/api/users',
  description: 'Get users with filters',
  queryParams: [
    { name: 'role', type: 'string', required: false },
    { name: 'page', type: 'number', required: false }
  ]
}
```

### Adding Request/Response Bodies

Add to the endpoint structure:

```typescript
endpoint: {
  method: 'POST',
  path: '/api/users',
  description: 'Create new user',
  requestBody: {
    type: 'json',
    example: { name: 'John', email: 'john@example.com' }
  }
}
```

## Best Practices

1. **Keep descriptions concise** - One sentence per service
2. **Use consistent naming** - Match your actual code naming
3. **Group related endpoints** - Organize by resource type
4. **Update regularly** - Keep documentation in sync with code
5. **Document dependencies** - Show service relationships clearly

## Tabs Overview

Each repository page has 4 main tabs:

### 📋 Services Tab
- Lists all business logic layers
- Shows service responsibilities
- Organized in a grid layout

### 🎮 Controllers Tab
- Shows which services each controller uses
- Lists all endpoints for each controller
- Color-coded HTTP methods

### 🔌 API Endpoints Tab
- Complete table of all endpoints
- Sortable by method, path, controller
- Quick reference for integration

### 🏗️ Architecture Tab
- Visual layer representation
- Shows data flow (Services → Controllers → Database)
- Helps understand system organization
